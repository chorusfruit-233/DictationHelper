# Android 个人听写辅助 App 需求说明

## 1. 项目一句话说明

我要开发一个 Android 个人学习工具。它可以通过拍照或导入图片，把英语课本上的单词表识别出来，整理成固定 JSON 词库；然后在听写练习时，通过麦克风听老师/家长/自己读出的中文、英文、提示语或控制语，实时在词库中找到最可能对应的英文单词或短语。

这个项目面向个人学习、课前课后复习、自测和错词整理，不用于考试作弊或违反课堂规则的场景。

---

## 2. 我的真实背景

我是零基础编程用户。

请 AI 在开发时遵守以下原则：

1. 不要默认我懂 Kotlin、Android、Gradle、Compose、Activity、Manifest、权限、数据库等概念。
2. 每一步都要告诉我：
   - 要打开哪个文件
   - 在哪个位置添加或替换代码
   - 替换前后大概是什么样
   - 完成后如何运行验证
3. 不要一次性给我太多文件和太复杂的架构。
4. 每次只做一个小目标，先保证项目能运行，再逐步加功能。
5. 如果报错，请根据报错信息逐行解释，不要只说“检查依赖”。
6. 代码要完整、可复制、能直接运行。
7. 优先使用官方、稳定、适合新手的方案。
8. 不要为了“架构漂亮”过早引入复杂设计模式。

### 3.1 平台

- Android App
- Android Studio 新建项目
- 模板选择：Empty Activity

## 5. 词表导入模块需求

### 5.1 用户目标

用户可以把书上的单词表拍照或导入图片，然后让 AI/OCR 识别内容，整理成结构化词库。

### 5.2 输入来源

第一版可以先不做真实拍照，先支持手动粘贴一段文本或导入示例 JSON。

后续再支持：

- 拍照
- 从相册选择图片
- OCR 识别
- 视觉 AI 直接理解图片中的表格结构

### 5.3 识别内容

书本单词表可能包含：

- 英文单词
- 英文短语
- 中文释义
- 音标
- 单元名
- 页码
- 词性
- 例句
- 多栏排版
- 不规则换行

### 5.4 重要原则

AI/OCR 不需要一次性 100% 正确。

正确流程应该是：

```text
图片/文本输入
↓
AI/OCR 初步识别
↓
转换成 JSON
↓
用户人工校对
↓
保存为词库
```

也就是说，识别结果必须给用户一个“校对页面”，不要直接静默保存。

## 6. 词库 JSON 格式

请使用下面的 JSON 结构作为项目初始标准。

```json
{
  "bookName": "English Book",
  "unitName": "Unit 1",
  "language": "en",
  "items": [
    {
      "id": "unit1_001",
      "type": "word",
      "text": "achievement",
      "normalizedText": "achievement",
      "meaningZh": "成就；成绩",
      "phonetic": "",
      "partOfSpeech": "n.",
      "aliases": [],
      "example": "",
      "tags": ["Unit 1"]
    },
    {
      "id": "unit1_002",
      "type": "phrase",
      "text": "take care of",
      "normalizedText": "take care of",
      "meaningZh": "照顾；照料",
      "phonetic": "",
      "partOfSpeech": "",
      "aliases": ["look after"],
      "example": "",
      "tags": ["Unit 1"]
    }
  ]
}
```

### 字段说明

| 字段 | 含义 |
|---|---|
| bookName | 书名 |
| unitName | 单元名 |
| language | 主要语言，英语用 en |
| items | 单词/短语列表 |
| id | 唯一 ID |
| type | word 或 phrase |
| text | 正确英文答案 |
| normalizedText | 标准化后的英文，用于匹配 |
| meaningZh | 中文释义 |
| phonetic | 音标，可为空 |
| partOfSpeech | 词性，可为空 |
| aliases | 近义表达或别名 |
| example | 例句，可为空 |
| tags | 标签，例如单元、页码、课本版本 |

## 7. 听写匹配模块需求

### 7.1 用户目标

用户选择某个单元词库后，打开听写模式。App 监听老师/家长/自己说的话，并实时判断当前可能对应哪个英文单词或短语。

### 7.2 真实听写场景

老师不一定只报英文单词。老师可能会说很多种混合指令。

例如：

```text
next one
下一个
中文是“照顾”
这个词是 t 开头
三个词
不是 care about，是另一个
第十五个
上一个
repeat
这个短语的意思是“照顾”
take... 后面是什么
```

所以这个功能不是单纯的“语音转文字”，而是：

```text
语音转文字
↓
理解老师这句话的意图
↓
提取约束条件
↓
根据本地词库匹配最可能的单词/短语
```

---

## 8. 大模型在本项目中的作用

大模型不应该直接“凭空猜答案”。

正确做法是：

```text
老师说的话
↓
ASR 得到文本
↓
大模型把文本解析成结构化 JSON 意图
↓
App 用本地词库确定性匹配
↓
显示候选结果
```

### 8.1 大模型输出格式

大模型应该尽量输出下面这种 JSON：

```json
{
  "action": "FIND_WORD",
  "rawText": "中文是照顾，三个词，t开头",
  "constraints": {
    "meaningZh": "照顾",
    "englishPartial": null,
    "firstLetter": "t",
    "wordCount": 3,
    "index": null
  },
  "shouldAdvance": false
}
```

### 8.2 action 类型

| action | 含义 |
|---|---|
| FIND_WORD | 根据线索查找单词或短语 |
| NEXT_ITEM | 下一个 |
| PREVIOUS_ITEM | 上一个 |
| REPEAT_CURRENT | 重复当前 |
| JUMP_TO_INDEX | 跳到第几个 |
| ADD_HINT | 给当前题增加提示 |
| REJECT_CANDIDATE | 排除某个候选 |
| UNKNOWN | 无法理解 |

### 8.3 constraints 字段

| 字段 | 含义 |
|---|---|
| meaningZh | 中文释义线索 |
| englishPartial | 英文片段，例如 take care |
| firstLetter | 首字母 |
| wordCount | 英文短语由几个词组成 |
| index | 第几个词 |
| rejectedText | 被排除的候选词 |
| rawHint | 原始提示 |

## 9. 本地匹配逻辑

App 应该主要依赖本地词库匹配，而不是完全依赖大模型。

### 9.1 匹配依据

匹配时可以使用以下线索：

1. 中文释义匹配
2. 英文完整匹配
3. 英文片段匹配
4. 首字母匹配
5. 短语词数匹配
6. 当前单元范围
7. 当前听写进度附近的词
8. 被排除候选
9. aliases 近义表达

### 9.2 示例

词库中有：

```json
{
  "text": "take care of",
  "meaningZh": "照顾；照料",
  "type": "phrase"
}
```

老师说：

```text
中文是照顾，三个词，t开头
```

大模型解析为：

```json
{
  "action": "FIND_WORD",
  "constraints": {
    "meaningZh": "照顾",
    "firstLetter": "t",
    "wordCount": 3
  }
}
```

本地匹配器应该返回：

```json
{
  "matchedText": "take care of",
  "confidence": 0.97,
  "reason": "中文释义、首字母、词数均匹配"
}
```

## 10. 听写状态机

App 需要维护当前听写状态。

### 10.1 状态内容

```json
{
  "currentUnit": "Unit 1",
  "currentIndex": 0,
  "currentCandidates": [],
  "confirmedItems": [],
  "rejectedCandidates": [],
  "recentTeacherUtterances": []
}
```

### 10.2 为什么需要状态机

因为老师可能会说：

```text
下一个
不是这个
再重复一遍
上一个
第十五个
这个词 a 开头
```

这些话必须结合“当前进行到哪里”才能理解。

## 11. 推荐 MVP 开发顺序

请严格按下面顺序开发，不要一上来就做完整最终版。

### MVP 0：项目能运行

目标：

- Android Studio 创建 Empty Activity 项目
- Kotlin + Jetpack Compose
- 首页显示 app 名称
- 有两个按钮：
  - 导入词表
  - 开始听写

验收标准：

- 点击 Run 能在模拟器或真机上打开 app
- 不崩溃
- 首页能看到两个按钮

---

### MVP 1：内置示例词库

目标：

- 暂时不做拍照和 OCR
- 在代码里内置一个示例词库
- 显示词库列表

示例词库：

```json
[
  {
    "id": "unit1_001",
    "type": "word",
    "text": "achievement",
    "meaningZh": "成就；成绩"
  },
  {
    "id": "unit1_002",
    "type": "phrase",
    "text": "take care of",
    "meaningZh": "照顾；照料"
  },
  {
    "id": "unit1_003",
    "type": "word",
    "text": "environment",
    "meaningZh": "环境"
  }
]
```

验收标准：

- App 能展示这些单词
- 每个条目显示英文和中文

---

### MVP 2：手动输入老师文本并匹配

目标：

- 暂时不接麦克风
- 用户手动输入一句“老师说的话”
- App 根据这句话匹配词库

输入示例：

```text
中文是照顾，三个词，t开头
```

输出示例：

```text
最可能答案：take care of
置信度：高
原因：中文、首字母、词数匹配
```

验收标准：

- 不用语音也能测试核心逻辑
- 至少支持中文释义、首字母、词数匹配

---

### MVP 3：加入简单规则解析器

目标：

先不用大模型，写一个简单规则解析器处理常见表达。

支持：

```text
下一个
上一个
中文是xxx
xxx开头
几个词
第几个
```

验收标准：

- 输入“中文是照顾”能找到 take care of
- 输入“三个词 t 开头 照顾”能找到 take care of
- 输入“下一个”能让 currentIndex + 1

---

### MVP 4：接入语音识别

目标：

- 使用 Android SpeechRecognizer
- 把老师说的话转成文本
- 将文本交给 MVP 3 的解析器和匹配器

验收标准：

- 点击“开始听”
- App 显示识别出的文字
- App 显示候选答案

---

### MVP 5：接入大模型语义解析

目标：

- 把 SpeechRecognizer 的文本交给大模型
- 大模型只负责输出结构化 JSON
- 本地匹配器仍然负责查词

验收标准：

- 对复杂句子理解更好
- 大模型不直接返回凭空答案
- 大模型输出必须能被 App 解析

---

### MVP 6：词表导入

目标：

- 支持粘贴 OCR 文本
- AI 整理成标准 JSON
- 用户确认后保存

后续再做：

- 拍照
- 相册导入
- OCR
- 视觉 AI 识别图片中的单词表

## 13. 非功能需求

### 13.1 新手友好

代码必须简单、清楚、可复制。

### 13.2 稳定优先

不要一开始就追求最先进方案。

### 13.3 本地优先

词库匹配尽量在本地完成。

### 13.4 可解释

每次匹配都要显示原因，比如：

```text
因为“照顾”匹配中文释义，“t”匹配首字母，“三个词”匹配短语长度。
```

### 13.5 可校对

AI 识别词表后必须允许用户修改，不能直接保存。

## 14. 安全和使用边界

这个 App 的定位是：

- 个人复习
- 课前预习
- 课后自测
- 错词整理
- 家长/自己模拟听写

不应该用于：

- 正式考试作弊
- 违反学校课堂规则
- 偷偷获取听写答案

如果 AI 在设计功能时涉及实时课堂使用，请默认把它设计成“练习模式”和“自测模式”。
